Beginner-Friendly Walkthrough

How Sentinel Scans Code

No security background? No problem. Read this simple guide to understand the metaphors and metrics behind our software security report.

The Four Core Metaphors

📦

The Binary file

Metaphor: The Locked Box of Recipes

A "binary" (like an .exe file) is a program compiled for computers. Think of it as a **locked wooden box** containing a recipe book. The computer can open the box and follow the recipes instantly, but a human cannot read the book directly because it is written in machine code. To see if it is safe, we need special tools to look inside.

🗺️

The Control Flow Graph

Metaphor: The Road Map

When the program runs, it makes choices (e.g., "If password is correct, go right; else, go left"). We map these choice points into a **Control Flow Graph (CFG)**. Think of this as a **road map** representing all intersections and highways inside the program. A vulnerability is like a road segment that ends in a cliff without a guard rail.

🕵️

The GNN Classifier

Metaphor: The Road Safety Inspector

A **Graph Neural Network (GNN)** is our custom AI model. Think of it as a **highly trained road inspector**. It looks at the Road Map we drew and checks the shape of the intersections. It doesn't read the code line-by-line; instead, it recognizes if the connections look like a dangerous road setup where an accident is likely to happen.

📘

The RAG Security Auditor

Metaphor: The Engineer with the Safety Manual

Once the GNN Inspector flags a dangerous intersection, the **Retrieval-Augmented Generation (RAG)** assistant steps in. Think of this as an **expert engineer** who opens the official **National Road Construction Safety Manual** (standard rules like CWE and CERT C), explains *why* the road is unsafe, and gives the builders a blueprint on how to rebuild it securely.

🚨 The Verdict Box
Metaphor: The Safety Flag

The **Verdict** is the final "Yes or No" outcome of the entire scan.

  • SAFE: The AI scanned the entire application map and found no dangerous roads. It is ready for deployment.
  • VULNERABLE: The AI found at least one function containing structure patterns that represent a known security threat. It means the code needs to be modified before releasing.

Analogy: Like a traffic light. Green means safe to go; Red means stop and fix the hazards.

📊 Threat Confidence
Metaphor: The Inspector's Certainty

The **Threat Confidence** represents how sure the AI model is of its classification verdict.

If the model classifies a function as vulnerable with **89.2% confidence**, it means the structural patterns inside the function align very strongly with known historical exploit shapes.

High confidence values require immediate engineering review, while low confidence scores (close to 50%) indicate potential border cases or minor anomalies.

🔍 Forensic Metadata
Metaphor: The Car's License Plate & Diagnostic Stats

This section lists identification details for the file under analysis:

  • Filename: The name of the file you uploaded.
  • SHA-256 Checksum: A unique, mathematically generated signature. Think of it as a digital fingerprint—even changing a single letter in the file changes this code completely, proving the report matches this exact file.
  • Evaluated Functions: The total count of logical tasks/functions inside the binary that the GNN inspector scanned.
🛠️ Secure Code Fix
Metaphor: The Road Reconstruction Blueprint

This is the most actionable part of the report. For every flagged vulnerability, the RAG assistant displays a block of **corrected source code**.

It shows developers exactly how to write the function securely (for example, replacing an unsafe, unchecked function like strcpy with a safe, bounded copy like strncat).

By implementing this fix, developers remove the weak spot and secure the application.

⚡ GCC/Clang Compiler Hardening Reference

Prevent exploitation of remaining buffer weaknesses by instructing the compiler to generate secure binary structures. Always include these flags during release compilation:

-fstack-protector-strong Stack Guard
Defends Against: Stack Smashing / Stack Buffer Overflow (CWE-121).
How it works: Places a secret random integer ("canary") on the stack frame right before the return pointer. If a buffer overflows, the canary is corrupted first. The runtime detects this and terminates execution before payload hijack occurs.
-D_FORTIFY_SOURCE=2 Safe Library Functions
Defends Against: Unchecked String Copies & Format Strings (CWE-120, CWE-134).
How it works: Replaces calls to vulnerable functions (like strcpy, memcpy) with compile-time and run-time size-checked equivalents (e.g. __strcpy_chk) if buffer sizes are statically known.
-fPIE -pie ASLR Compatibility
Defends Against: Return-Oriented Programming (ROP) Gadget Chains.
How it works: Compiles the executable as position-independent, allowing the OS kernel to randomize the code section load address (Address Space Layout Randomization). Attackers cannot guess gadget memory locations.
-Wl,-z,relro,-z,now Full RELRO
Defends Against: Global Offset Table (GOT) Overwrites.
How it works: Marks the Global Offset Table as read-only after resolving external function pointers at load time. Attackers cannot overwrite library calls (like redirecting printf to system).
