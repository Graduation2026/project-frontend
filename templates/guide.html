Beginner-Friendly Walkthrough

Understanding Your Security Report

Never heard of a "vulnerability" or "binary analysis"? That's fine. This guide assumes zero prior knowledge and walks you through what Sentinel does and how to read every part of your report.

Start Here: The 30-Second Version

What is a "vulnerability"? It is a mistake in a program that an attacker could abuse to crash it, steal data, or take control. Think of it as an unlocked window in an otherwise secure house. The program still works normally, so you usually cannot see the problem just by running it.

What does Sentinel do? You upload a program (or its source code). Sentinel pulls it apart, draws a map of how it works, and uses an AI trained on thousands of examples to check each part for these "unlocked windows." Then a second AI writes a plain-English report explaining what it found and how to fix it.

What do you get? A headline telling you how many parts were risky (for example, "1 out of 2 functions were vulnerable"), a list of each part marked Safe or Vulnerable, and a detailed report with fixes. The rest of this page explains every piece of that report.

One word you'll see everywhere: "function." A function is a single named task inside a program, like one worker with one job. A program is built from many functions working together (for example, main starts the program, process_user might handle a login). Sentinel checks each function separately, which is why the report talks about "X out of N functions."

How It Works: Four Simple Metaphors

Sentinel works in four stages. Here is each one explained with an everyday comparison.

📦

The Binary file

Metaphor: The Locked Box of Recipes

A "binary" (like an .exe file) is a program compiled for computers. Think of it as a **locked wooden box** containing a recipe book. The computer can open the box and follow the recipes instantly, but a human cannot read the book directly because it is written in machine code. To see if it is safe, we need special tools to look inside.

🗺️

The Control Flow Graph

Metaphor: The Road Map

When the program runs, it makes choices (e.g., "If password is correct, go right; else, go left"). We map these choice points into a **Control Flow Graph (CFG)**. Think of this as a **road map** representing all intersections and highways inside the program. A vulnerability is like a road segment that ends in a cliff without a guard rail.

🕵️

The GNN Classifier

Metaphor: The Road Safety Inspector

A **Graph Neural Network (GNN)** is our custom AI model. Think of it as a **highly trained road inspector**. It looks at the Road Map we drew and checks the shape of the intersections. It doesn't read the code line-by-line; instead, it recognizes if the connections look like a dangerous road setup where an accident is likely to happen.

📘

The RAG Security Auditor

Metaphor: The Engineer with the Safety Manual

Once the GNN Inspector flags a dangerous intersection, the **Retrieval-Augmented Generation (RAG)** assistant steps in. Think of this as an **expert engineer** who opens the official **National Road Construction Safety Manual** (standard rules like CWE and CERT C), explains *why* the road is unsafe, and gives the builders a blueprint on how to rebuild it securely.

Read Your Report, Top to Bottom

Your report appears in this order. Follow along from top to bottom — each step below matches a block you will actually see on screen.

1

The Headline

The big line at the top, for example "1 out of 2 functions were vulnerable." This is the quick answer. The first number is how many functions look risky; the second is how many of your real functions were checked. 0 out of N means nothing risky was found. Anything above 0 means at least one part needs your attention below.

2

The Three Counters

Three boxes summarise the scan:

  • Total Functions Evaluated — everything found inside the file.
  • Actual Code Functions Scanned — the functions you wrote, which the AI actually analysed.
  • Boilerplate & Runtime Stubs Found — automatic setup code added by the compiler that you did not write. Sentinel skips these so they don't cause false alarms.
3

Forensics (File Details)

Identification details so you know exactly which file this report describes: the filename, a unique fingerprint called the SHA-256 (change one byte of the file and this code changes completely), and timing stats showing how long each stage took. These do not indicate danger — they are for record-keeping.

4

The Function List (Category B)

The heart of the report. Every function you wrote is listed with a badge:

  • ✅ Safe — the AI did not find a known dangerous pattern in this function.
  • ⚠️ Vulnerable — the AI recognised a risky pattern. A CWE code (explained further down) names the type of weakness.

Focus your effort here on the functions marked Vulnerable.

5

The AI Security Report

A written report generated by AI. It contains, in order:

  • Audit Metadata — the same file details and result summary, in writing.
  • Risk Level — a colour rating (Green/Yellow/Orange/Red) based on what share of functions were risky.
  • Executive Summary — a short paragraph summarising the result in plain words.
  • Category B details — each function with its weakness, root cause, and a direction for the fix.
  • General Mitigations — protective compiler settings worth turning on regardless.
  • Category A — the skipped boilerplate, listed for completeness.

You can also download this as a PDF using the button above the report.

🎯 The Result Headline
Metaphor: The Scoreboard

The headline is the quick answer at the very top, written as "X out of N functions were vulnerable."

  • X = how many of your functions look risky.
  • N = how many of your real functions were checked.

0 out of N means nothing risky was found. Any number above 0 means you have functions to review in the list below.

Analogy: Like a scoreboard — it tells you the result instantly, then the rest of the report shows the play-by-play.

📋 The Function List
Metaphor: The Health Check-up Results

Every function you wrote gets its own row with a badge, like results from a medical check-up where each organ is checked separately:

  • ✅ Safe: No known dangerous pattern was found in this function.
  • ⚠️ Vulnerable: A risky pattern was recognised. A CWE code names the weakness type (see the CWE section below).

Spend your time on the functions marked Vulnerable — those are the ones that may need fixing.

📊 Risk Level
Metaphor: The Weather Warning System

The Risk Level is a color-coded severity tier based on the percentage of functions flagged as vulnerable:

  • GREEN (Low): 0% — No vulnerabilities detected. All clear.
  • YELLOW (Medium): 1-25% — Some functions flagged. Review recommended.
  • ORANGE (High): 26-50% — Significant threat surface. Remediation needed before deployment.
  • RED (Critical): 51%+ — Majority of functions are vulnerable. Immediate intervention required.

Analogy: Like a weather warning system — Green is clear skies, Yellow is watch advisory, Orange is a warning, and Red is a severe emergency alert.

🔍 Forensics
Metaphor: The File's ID Card & Receipt

Identification details so you know exactly which file this report describes. These do not indicate danger:

  • Filename: The name of the file you uploaded.
  • SHA-256 Checksum: A unique digital fingerprint. Change a single byte of the file and this code changes completely, proving the report matches this exact file.
  • Decompile / Inference / Total Time: How long each stage took — useful context, like a receipt showing the work done.
🛠️ Fix Direction
Metaphor: The Repair Instructions

For every flagged function, the AI report gives a short, practical breakdown:

  • Threat ID: the CWE weakness type.
  • Root Cause: why it is risky in one line.
  • CERT C Rule Violated: the official coding rule it breaks.
  • Fix Direction: a one-line pointer on how to repair it.

Need a full corrected code example? Open the in-app chat assistant and ask it about the flagged function — it can walk you through a secure rewrite.

🗂️ Category A vs B
Metaphor: The Packaging vs The Product

When you compile a program, the compiler injects a lot of "invisible" setup code (like _init or _start_c) that you never wrote. Sentinel AI separates these so they don't trigger false alarms.

  • Category A (Boilerplate): The "packaging" (startup stubs, library wrappers). These are filtered out and ignored by the vulnerability scanner.
  • Category B (Actual Code): The "product" (the actual logic functions you wrote like main or process_user). These are heavily analyzed.

What Is a "CWE"?

When a function is flagged, the report attaches a code like CWE-119 or CWE-787. CWE stands for "Common Weakness Enumeration" — a public, industry-standard catalogue of software weakness types maintained by the security community. Think of it like a medical diagnosis code: instead of "something is wrong," it tells you exactly which known problem it is, so anyone in the world can look it up and understand it the same way.

A CWE does not mean your program was hacked. It means this function contains a pattern that is known to be dangerous if left unfixed. A few you may see:

  • CWE-119 / CWE-787 (Memory / Out-of-bounds Write) — the program may write data outside the space reserved for it, like pouring more water than a glass can hold.
  • CWE-120 (Buffer Overflow / unchecked copy) — copying data without checking if it fits.
  • CWE-134 (Format String) — letting untrusted input control how text is formatted.

You can search any code at cwe.mitre.org for a full description and real-world examples.

⚡ GCC/Clang Compiler Hardening Reference

Prevent exploitation of remaining buffer weaknesses by instructing the compiler to generate secure binary structures. Always include these flags during release compilation:

-fstack-protector-strong Stack Guard
Defends Against: Stack Smashing / Stack Buffer Overflow (CWE-121).
How it works: Places a secret random integer ("canary") on the stack frame right before the return pointer. If a buffer overflows, the canary is corrupted first. The runtime detects this and terminates execution before payload hijack occurs.
-D_FORTIFY_SOURCE=2 Safe Library Functions
Defends Against: Unchecked String Copies & Format Strings (CWE-120, CWE-134).
How it works: Replaces calls to vulnerable functions (like strcpy, memcpy) with compile-time and run-time size-checked equivalents (e.g. __strcpy_chk) if buffer sizes are statically known.
-fPIE -pie ASLR Compatibility
Defends Against: Return-Oriented Programming (ROP) Gadget Chains.
How it works: Compiles the executable as position-independent, allowing the OS kernel to randomize the code section load address (Address Space Layout Randomization). Attackers cannot guess gadget memory locations.
-Wl,-z,relro,-z,now Full RELRO
Defends Against: Global Offset Table (GOT) Overwrites.
How it works: Marks the Global Offset Table as read-only after resolving external function pointers at load time. Attackers cannot overwrite library calls (like redirecting printf to system).

What Do I Do Now?

A simple plan once your report is ready.

1️⃣

Check the headline

If it says 0 out of N, no risky functions were found — you can move on. If it is above 0, continue below.

2️⃣

Open each Vulnerable function

In the function list, note which ones are marked Vulnerable and read their CWE and Fix Direction in the AI report.

3️⃣

Ask the chat assistant

Not sure how to fix it? Use the in-app chat to ask for a secure rewrite of that specific function in plain language.

4️⃣

Harden & re-scan

Apply the fix, enable the compiler hardening flags above, rebuild, and upload again to confirm the headline improved.

Quick Glossary

Binary — a compiled program a computer can run (e.g. .exe or .o), not readable by humans directly.

Function — one named task inside a program. Sentinel checks each separately.

Vulnerability — a coding mistake an attacker could abuse.

CWE — a standard code naming the type of weakness found.

CFG (Control Flow Graph) — a map of all the paths a program can take while running.

GNN — the AI "inspector" that studies the map and spots dangerous shapes.

Boilerplate / Runtime Stub — automatic setup code added by the compiler; skipped by the scanner.

SHA-256 — a unique fingerprint that identifies your exact file.

Risk Level — a Green/Yellow/Orange/Red rating based on the share of functions flagged.
